# Offline PWA and Push Notifications

## 1. PWA Configuration

**`manifest.json` fields:**

| Field | Value |
|-------|-------|
| `name` | `APP_NAME` env var (injected at build time via `VITE_APP_NAME`) |
| `short_name` | Same as `name` |
| `start_url` | `/` |
| `display` | `standalone` |
| `background_color` | `#ffffff` |
| `theme_color` | `APP_THEME_COLOR` env var (default `#6366f1`), injected as `VITE_APP_THEME_COLOR` |
| `orientation` | `any` |
| `icons` | See below |

**Required icon sizes:**
- 192×192 PNG — standard icon
- 512×512 PNG — splash screen / install prompt
- 192×192 maskable PNG — Android adaptive icon (include `"purpose": "maskable"`)
- 512×512 maskable PNG

Icons live in `frontend/public/icons/`. Filenames: `icon-192.png`, `icon-512.png`, `icon-192-maskable.png`, `icon-512-maskable.png`.

**iOS considerations:**
- Add `<meta name="apple-mobile-web-app-capable" content="yes">` in `index.html`
- Add `<link rel="apple-touch-icon" href="/icons/icon-192.png">` in `index.html`
- iOS requires the app to be added to the Home Screen for push notifications (iOS 16.4+)

## 2. Service Worker

**Tooling:** `vite-plugin-pwa` (Workbox-based). Configuration in `vite.config.ts`.

**Caching strategies:**

| Asset Type | Strategy | Notes |
|-----------|---------|-------|
| App shell (JS/CSS bundles) | `CacheFirst` | Bust by build hash in filename |
| API responses (`/api/*`) | `NetworkFirst` | 5-second timeout; fall back to cache on timeout/failure |
| Static assets (`/icons/*`, `/images/*`) | `CacheFirst` | Long TTL |
| Navigation requests | `NetworkFirst` | Fall back to `index.html` for offline SPA |

The Service Worker is auto-generated by Workbox from the `vite-plugin-pwa` configuration. Manual SW code (push event handler, background sync) is added via `injectManifest` mode or a custom SW file.

## 3. Offline File Upload Queue — IndexedDB Schema

**Database name:** `homekm-offline-queue`
**Version:** 1

**Object store: `upload-queue`**

```typescript
interface QueuedUpload {
  clientUploadId: string;      // UUID, keyPath (primary key)
  file: Blob;                  // File content stored locally
  filename: string;
  mimeType: string;
  sizeBytes: number;
  folderId: number | null;
  description: string | null;
  isChildSafe: boolean;
  tagIds: number[];
  status: 'pending' | 'uploading' | 'done' | 'failed';
  createdAt: number;           // epoch milliseconds
  retryCount: number;
  lastError: string | null;
}
```

**Object store: `auth`**

```typescript
// Single record, key = 'token'
interface AuthRecord {
  key: 'token';
  value: string;               // JWT string
  expiresAt: number;           // epoch milliseconds
}
```

The main React app writes to `auth` whenever the JWT is set or refreshed. The Service Worker reads from `auth` to include the `Authorization` header in upload requests.

**Library:** Use the `idb` npm package for typed IndexedDB access.

## 4. Sync Protocol

### Triggering a sync

When a file is added to the `upload-queue` store:
1. Set the record's `status` to `pending`
2. Register a Background Sync event: `navigator.serviceWorker.ready.then(sw => sw.sync.register('file-upload-sync'))`
3. If Background Sync API is not supported (Safari < 16.4), fall back to immediate fetch attempt in the main thread

### Service Worker sync handler

```javascript
self.addEventListener('sync', event => {
  if (event.tag === 'file-upload-sync') {
    event.waitUntil(processPendingUploads());
  }
});
```

### `processPendingUploads()` logic

```
1. Open IndexedDB
2. Read all records from 'upload-queue' where status = 'pending'
3. Read JWT from 'auth' store
4. For each pending record:
   a. Set status = 'uploading'
   b. Build multipart/form-data with: file, folderId, description, isChildSafe, tagIds, clientUploadId
   c. POST to /api/files/upload with Authorization header
   d. On 2xx response:
      - Set status = 'done'
      - Optionally delete from IndexedDB (or retain for 24h audit)
   e. On 4xx (non-retryable) response:
      - Set status = 'failed', lastError = response body, retryCount++
   f. On 5xx or network error:
      - Increment retryCount
      - If retryCount >= 3: set status = 'failed'
      - Else: set status = 'pending' (will retry next sync)
5. After processing all, post message to main thread with summary
```

### Background Sync browser compatibility

| Browser | Background Sync |
|---------|----------------|
| Chrome / Edge | Supported |
| Firefox | Not supported (fall back to online event listener) |
| Safari | iOS 16.4+ supported |

Implement the `online` event fallback for browsers without Background Sync:
```javascript
window.addEventListener('online', () => {
  navigator.serviceWorker.ready.then(sw => sw.sync.register('file-upload-sync'));
});
```

## 5. Connectivity Detection and UI

The main app registers `online` / `offline` event listeners:

```javascript
window.addEventListener('online', () => setIsOnline(true));
window.addEventListener('offline', () => setIsOnline(false));
```

Initial state: `navigator.onLine`.

**Offline banner:** When `isOnline = false`, display a persistent banner at the top of the screen: "Offline — file uploads will sync when connection is restored."

**Queue indicator:** When `pendingUploadCount > 0` (read from IndexedDB), show a badge on the file upload button. Show individual queued items in the upload panel with their status.

**Failed uploads UI:** Items with `status = 'failed'` and `retryCount >= 3` show an error indicator and a "Retry" button. Clicking Retry resets `status` to `pending` and `retryCount` to `0`, then re-registers the sync tag.

## 6. Web Push Subscription Flow

### Initial setup (on app load)

```javascript
1. Check Notification.permission
2. If 'default':
   - Show an in-app prompt card (not the browser native dialog yet):
     "Enable push notifications to receive reminders"
     [Enable] [Not now]
3. On user clicking [Enable]:
   a. Call Notification.requestPermission()
   b. If granted:
      - Fetch VAPID public key from GET /api/vapid-public-key (cache in sessionStorage)
      - Call navigator.serviceWorker.ready
      - Call registration.pushManager.subscribe({
          userVisibleOnly: true,
          applicationServerKey: vapidPublicKey  // base64url decoded to Uint8Array
        })
      - POST the PushSubscription object to POST /api/push/subscribe
4. If permission = 'denied': show static info "Enable in browser settings to receive reminders"
5. If permission = 'granted': check if already subscribed; if not, re-subscribe
```

### iOS-specific flow

Detect iOS: `navigator.userAgent.match(/iPhone|iPad/)` and `!window.navigator.standalone`.

If iOS and not standalone: show persistent banner "Add this app to your Home Screen to enable push notifications."

### Push event handler (in Service Worker)

```javascript
self.addEventListener('push', event => {
  const data = event.data ? event.data.json() : {};
  event.waitUntil(
    self.registration.showNotification(data.title || 'Home KM', {
      body: data.body,
      icon: data.icon || '/icons/icon-192.png',
      badge: '/icons/icon-72.png',
      data: data.data || {}
    })
  );
});

self.addEventListener('notificationclick', event => {
  event.notification.close();
  const noteId = event.notification.data.noteId;
  const url = noteId ? `/notes/${noteId}` : '/';
  event.waitUntil(
    clients.matchAll({ type: 'window' }).then(windowClients => {
      const existing = windowClients.find(c => c.url.includes(url));
      if (existing) return existing.focus();
      return clients.openWindow(url);
    })
  );
});
```

## 7. PWA Install Prompt

Detect the `beforeinstallprompt` event and store it. Show an in-app "Install App" button in the settings or header when the event is available. Do not auto-trigger the native install dialog.

```javascript
let deferredPrompt;
window.addEventListener('beforeinstallprompt', e => {
  e.preventDefault();
  deferredPrompt = e;
  setInstallable(true);
});
```

On button click: `deferredPrompt.prompt()`.
