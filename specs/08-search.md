# Search

## 1. Search Endpoint

### `GET /api/search`

**Auth required:** Yes

**Query parameters:**

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `q` | String | Yes | — | Search query, 1–500 chars (after trimming) |
| `types` | String[] | No | all | Comma-separated subset of `note`, `file`, `folder` |
| `folderId` | Long | No | — | Restrict results to this folder and all descendants |
| `tagIds` | Long[] | No | — | AND filter: results must have all specified tags |
| `page` | Integer | No | `0` | Zero-based page index |
| `size` | Integer | No | `20` | Max `50` |

**Validation:**
- `q` after trimming must be at least 1 character; `400 QUERY_REQUIRED` if empty or whitespace-only
- `q` max 500 characters; `400 VALIDATION_ERROR` if exceeded
- `size` max 50; capped silently (no error) if exceeded
- `types` values must each be `note`, `file`, or `folder`; `400 INVALID_TYPE` for unknown values

```json
    {

      "id": 142,

```

- `excerpt` — snippet generated by PostgreSQL `ts_headline()`, max 200 chars, HTML `<b>` tags highlight matched terms

- `label` — only present when `type = "note"`
- `mimeType` — only present when `type = "file"`

## 3. PostgreSQL FTS Implementation

Use `plainto_tsquery('english', :query)` for all user input. This is safe against injection and handles common stop words. Do not use `to_tsquery()` which requires the user to understand tsquery syntax.

**Core UNION query:**

```sql
WITH query AS (
    SELECT plainto_tsquery('english', :query) AS q
)
SELECT
    'note'              AS type,
    n.id,
    n.title,
    ts_headline('english', COALESCE(n.body, ''), q.q,
        'MaxWords=30, MinWords=15, MaxFragments=1') AS excerpt,
    ts_rank_cd(n.search_vector, q.q) AS score,
    n.is_child_safe,
    n.folder_id,
    f.name AS folder_name,
    n.label,
    NULL::VARCHAR AS mime_type,
    n.updated_at
FROM notes n
CROSS JOIN query q
LEFT JOIN folders f ON n.folder_id = f.id
WHERE n.search_vector @@ q.q
  AND <child_safe_filter>
  AND <folder_scope_filter>
  AND <tag_filter>

UNION ALL

SELECT
    'file'              AS type,
    fi.id,
    fi.filename         AS title,
    ts_headline('english', COALESCE(fi.description, ''), q.q,
        'MaxWords=30, MinWords=15, MaxFragments=1') AS excerpt,
    ts_rank_cd(fi.search_vector, q.q) AS score,
    fi.is_child_safe,
    fi.folder_id,
    f.name AS folder_name,
    NULL   AS label,
    fi.mime_type,
    fi.updated_at
FROM files fi
CROSS JOIN query q
LEFT JOIN folders f ON fi.folder_id = f.id
WHERE fi.search_vector @@ q.q
  AND <child_safe_filter>
  AND <folder_scope_filter>
  AND <tag_filter>

UNION ALL

SELECT
    'folder'            AS type,
    fo.id,
    fo.name             AS title,
    ts_headline('english', COALESCE(fo.description, ''), q.q,
        'MaxWords=30, MinWords=15, MaxFragments=1') AS excerpt,
    ts_rank_cd(fo.search_vector, q.q) AS score,
    fo.is_child_safe,
    fo.parent_id        AS folder_id,
    pf.name             AS folder_name,
    NULL                AS label,
    NULL                AS mime_type,
    fo.updated_at
FROM folders fo
CROSS JOIN query q
LEFT JOIN folders pf ON fo.parent_id = pf.id
WHERE fo.search_vector @@ q.q
  AND <child_safe_filter>
  AND <folder_scope_filter_for_folder>

ORDER BY score DESC
LIMIT :size OFFSET :page * :size
```

For `totalCount`, run a separate COUNT query using the same WHERE clauses without LIMIT/OFFSET.

## 4. Child-Safe Filter

Applied to every branch of the UNION when the requesting user has `is_child = true`:

```sql
AND <table>.is_child_safe = TRUE
```

For adult users, this clause is omitted entirely (not `AND TRUE` — just absent from the query).

The child-safe filter is applied in the query builder, not in application code post-fetch.

---

## 5. Folder Scope Filter

Applied when `folderId` parameter is provided. Restricts results to entities whose `folder_id` is in the set of descendant folder IDs (including the target folder itself).

**Recursive CTE to compute descendant IDs:**
```sql
WITH RECURSIVE subtree AS (
    SELECT id FROM folders WHERE id = :folderId
    UNION ALL
    SELECT f.id FROM folders f
    JOIN subtree s ON f.parent_id = s.id
)
```

Then apply:
```sql
-- For notes and files:
AND <table>.folder_id IN (SELECT id FROM subtree)

-- For folders:
AND fo.id IN (SELECT id FROM subtree)
```

Maximum depth for subtree traversal: 20 levels. If the scoped folder has more than 20 levels of descendants, results beyond level 20 are silently excluded (consistent with folder depth limit).

---

## 6. Tag Filter

Applied when `tagIds` parameter is provided. Results must have **all** specified tags (AND semantics).

**Subquery approach for notes:**
```sql
AND n.id IN (
    SELECT entity_id FROM taggings
    WHERE entity_type = 'note'
      AND tag_id IN (:tagIds)
    GROUP BY entity_id
    HAVING COUNT(DISTINCT tag_id) = :tagCount
)
```

Same pattern for `files` (entity_type = 'file') and `folders` (entity_type = 'folder').

Where `:tagCount` is the number of IDs in the `tagIds` parameter.

---

## 7. Type Filter

When `types` parameter is provided, exclude the corresponding UNION branches entirely. For example, `types=note,file` omits the folders UNION ALL branch from the query.

This is implemented as query builder logic, not a WHERE clause — unused branches are not appended to the SQL string.

---

## 8. Performance Expectations

For a household-scale dataset (up to 10,000 total notes + files + folders):
- GIN indexes on `search_vector` columns must be in place (defined in `01-data-model.md`)
- Search results must return in under 500ms for typical queries
- `EXPLAIN ANALYZE` must be run and documented during integration testing
- Folder scope with deep trees (>10 levels) may be slower — the 20-level cap mitigates the worst case

---

## 9. Input Sanitisation

- `q` is passed only through `plainto_tsquery()` — never string-interpolated into SQL
- All other parameters are passed as named JDBC parameters (`:folderId`, `:tagIds`, etc.)
- No SQL injection surface
